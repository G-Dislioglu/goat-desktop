# GOAT Desktop App Goal

Updated: 2026-05-29
Baseline: HEAD after GOAT-RECOVERY-001

GOAT Desktop is a local Windows assistant for normal users. It should help the user understand the visible desktop, mark safe targets, and perform only narrow mouse or keyboard actions after local verification and explicit user approval.

Builder is a proposal source, not an execution authority. GOAT Desktop owns the local trust boundary:

- local geometry verification in `broker.py`
- visible popup approval in `tray.py` / `popup.py`
- staged action gating in `action_gate.py`
- narrow executors for Stage 1 and Stage 2 only
- review-only handling for Stage 3
- technical lock for Stage 4

## Product Priorities

1. Ship visible user progress before polishing internal wording again.
2. Make the assistant clear for normal users.
3. Preserve user control before any real mouse or keyboard action.
4. Keep action semantics simple: check target, then either execute safe action or explain why not.
5. Prefer controlled, local, auditable behavior over broad autonomy.
6. Keep LiveTalk secondary unless explicitly requested.

## Product Progress Gate

Every implementation run must end with one of these:

- a visible local capability the user can exercise
- a live or controlled acceptance artifact proving the path that changed
- a packaging/startup improvement that makes GOAT easier to run
- a narrowly scoped safety fix tied to a named bug or failing test

Do not spend another run only on copy, redaction, or wording unless the run names the bug, failing test, or recovery contract it fixes. The next product proof is GOAT-LIVE-001: Builder cue reaches Desktop and produces a visible popup proposal with no desktop effects before user approval.

## Non-Goals

- No cloud deployment.
- No trading, purchasing, deleting, sending, saving, deploying, or similar consequential OS action.
- No automatic interaction with secrets, passwords, tokens, 2FA, CVV, PIN, or recovery codes.
- No Vision-only coordinate authority.
- No tests on real user fields.

## Current Safe Action Model

- Stage 1: hover, move, scroll. User approval required. No click, no typing.
- Stage 2: one-line text input. User approval and `safe_text_context=true` required.
- Stage 3: important action review in popup. No OS execution.
- Stage 4: technical lock. User handles manually.
